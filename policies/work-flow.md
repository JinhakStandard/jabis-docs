# JABIS 표준 작업 플로우

> JABIS 프로젝트에서 기능 개발/수정 작업을 수행할 때 따르는 표준 플로우입니다.
> TDD 기반으로 **테스트 기준을 먼저 정의**한 후 구현에 들어갑니다.

---

## 1. 표준 플로우

```
1. 테스트 정의    — 이번 작업에서 검증할 항목을 먼저 정한다
2. 계획 수립      — 구현 방향과 영향 범위를 파악한다
3. 코드 구현      — 계획에 따라 코드를 작성한다
4. 빌드           — pnpm build:preview (미리보기 빌드)
5. 미리보기 테스트 — ai-e2e-tester로 E2E 테스트
6. 커밋 & push    — 테스트 통과 시
7. 운영 테스트     — 배포 후 운영 환경에서 E2E 테스트
```

---

## 2. 각 단계 상세

### 2.1 테스트 정의

작업 요청을 받으면 **코드를 작성하기 전에** 검증 항목을 먼저 정한다.

- 이번 작업으로 **무엇이 동작해야 하는지** 명확히 한다
- E2E 테스트에서 확인할 구체적인 시나리오를 정의한다
- 예: "메뉴에 '결재' 항목이 표시되고, 클릭 시 결재 목록 페이지로 이동"

### 2.2 계획 수립

구현 방향을 결정하고 영향 범위를 파악한다.

- 수정할 파일과 컴포넌트 식별
- 기존 코드 패턴 확인
- 다른 프로젝트에 영향이 있는지 확인 (shared-pages, @jabis/menu 등)

### 2.3 코드 구현

계획에 따라 코드를 작성한다.

- 기존 패턴을 따른다
- jabis-docs 관련 문서를 참조한다
- 변경 범위를 최소화한다

### 2.4 빌드

미리보기 빌드를 실행한다.

```bash
pnpm build:preview
```

- 반드시 `build:preview`를 사용한다 (`pnpm build`는 운영용)
- 빌드 실패 시 에러를 수정하고 재빌드한다

### 2.5 미리보기 테스트

ai-e2e-tester로 E2E 테스트를 실행한다.

- 2.1에서 정의한 검증 항목을 기반으로 테스트 프롬프트를 작성한다
- 대상 URL: `https://jabis-maker.ngrok-free.dev/preview/{프로젝트명}/`
- 상세: `policies/e2e-testing.md` 2절 (미리보기 테스트)

**실패 시:**
1. 원인 분석 → 수정 → 재빌드 → 재테스트
2. 최대 2회 재시도, 그래도 실패하면 사용자에게 보고

### 2.6 커밋 & push

미리보기 테스트 통과 후 커밋하고 push한다.

- `git status -u`로 전체 파일 확인 (누락 방지)
- feature 브랜치에 push → PR 생성
- 상세: `policies/git-workflow.md`

### 2.7 운영 테스트

배포 완료 후 운영 환경에서 E2E 테스트를 실행한다.

- 대상 URL: 운영 URL (`*.jinhakapply.com`)
- 상세: `policies/e2e-testing.md` 3절 (운영 테스트)

## 3. 플로우 다이어그램

```
작업 요청
  │
  ├─ 1. 테스트 정의 (무엇을 검증할 것인가?)
  │
  ├─ 2. 계획 수립 (어떻게 구현할 것인가?)
  │
  ├─ 3. 코드 구현
  │
  ├─ 4. pnpm build:preview
  │     │
  │     └─ 빌드 실패 → 수정 → 재빌드
  │
  ├─ 5. 미리보기 테스트 (ai-e2e-tester)
  │     │
  │     ├─ passed → 6으로
  │     └─ failed → 수정 → 재빌드 → 재테스트 (최대 2회)
  │           └─ 2회 실패 → 사용자에게 보고
  │
  ├─ 6. 커밋 & push (feature 브랜치 → PR)
  │
  ├─ 7. 배포 (PR merge → Pipeline → K3S)
  │
  └─ 8. 운영 테스트 (ai-e2e-tester)
        │
        ├─ passed → 완료
        └─ failed → 원인 분석 → 수정 → 1부터 반복
```
